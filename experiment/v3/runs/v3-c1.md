# Protocol — proj-m3 (`tidymark`)

## 1. What is this project?

`tidymark` is a small, single-purpose Python CLI that tidies Markdown files in place. Three normalizations:

| Function | Status per NOTES.md | Reality in code |
|---|---|---|
| `normalize_whitespace` | done + tested | `line.rstrip()` per line, straightforward |
| `normalize_bullets` | "done", tests pass | **known bug**, see below |
| `fix_headings` | deliberately parked | runs, but `base_level` hard-coded |

Structure: `normalize.py` (pure functions, no I/O), `tidymark.py` (argparse CLI, reads file → `tidy()` → writes file back), `tests/test_normalize.py` (2 tests, both happy-path). `NOTES.md` is an append-only running log by the owner and is the real source of intent — the README is only a usage blurb.

Note: the CLI does `from normalize import ...` (flat import, no package), so it only runs from the repo root; `pytest` picks it up via rootdir insertion. That is a latent packaging wrinkle, not the current priority.

## 2. Single most valuable next action

**Fix `normalize_bullets` so the `*`/`+` replacement is anchored to the start of a line (after optional indentation), and add a regression test containing inline `*emphasis*`.**

Why this one:

- The owner named it explicitly and ranked it: NOTES.md 2026-05-14 says it "is a genuine bug and it sits inside code we already treat as done, which makes it the thing anyone continuing here should fix first." That is an unambiguous priority statement, not an inference on my part.
- It is a **silent data-corruption bug in a tool that overwrites files in place** (`tidymark.py` writes back to the same path, no backup, no `--check` mode). `"an *important* note"` becomes `"an -important- note"`. `+ ` in prose ("C+ + grade", "1 + 2") is hit the same way. There is no undo except version control.
- The two existing tests pass, so the green suite actively hides the defect — exactly the failure mode a regression test closes.
- It is small and self-contained: one function, one added test, no design decision pending, no dependency needed.

The competing candidate — finishing the `# TODO: skip fenced code blocks` in `fix_headings` — looks like the obvious loose end because it is the only TODO in the code. **Do not take it.** NOTES.md 2026-05-20 says heading work was stopped on purpose: `base_level` is hard-coded to `1`, which silently rewrites docs for teams starting at `##`, and the owner wants it left alone until `base_level` becomes a documented config option or CLI flag. That is a blocked design decision belonging to the owner, not a coding gap.

## 3. Constraints a contributor must respect

1. **Standard library only. No third-party dependencies at all, and specifically no Markdown parsing library** — NOTES.md 2026-06-01. Reason given: must run on bare Python in CI images and on air-gapped machines. This directly constrains the bullet fix: use `re` (stdlib) or manual line handling, not `markdown-it`/`mistune`/`commonmark`.
2. **Do not touch heading normalization / `fix_headings`** until the `base_level` config question is decided by the owner. The in-code TODO is inside that frozen area. Deciding the config format (file-level vs CLI flag) is an owner call, not a contributor call.
3. **`normalize.py` holds pure functions only** — the module docstring states this, and all file I/O lives in `tidymark.py`. Keep the split.
4. **The tool rewrites files in place**, so any change to normalization semantics is destructive to users' documents on the next run. Regression tests are not optional here.
5. Not a hard rule but the owner's stated pattern: NOTES.md is append-only with dated entries ("I just append"). A contributor should add a dated entry rather than restructure the file.

Deferred wishlist, explicitly "someday, if there is time" (NOTES.md 2026-06-03): config-file support and a `--check` mode that only reports. Not next actions.

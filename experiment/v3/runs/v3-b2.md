# Protocol — proj-k7 (`tidymark`)

## 1. What is this project?

`tidymark` is a very small, self-contained Python CLI that tidies Markdown files in place. Three normalizers:

- `normalize_whitespace` — strips trailing whitespace per line (done, tested)
- `normalize_bullets` — rewrites `* `/`+ ` list bullets to `- ` (done, happy-path tested, buggy)
- `fix_headings` — shifts all ATX heading levels so the first heading sits at `base_level` (code runs, exposed via `--headings`, but **explicitly parked**)

`tidymark.py` is the argparse wrapper (`tidy()` composes the normalizers, writes the file back in place). `tests/test_normalize.py` holds two tests.

The repo is deliberately built to the "Readable Repo Standard" (per `AGENTS.md`): a fresh model docks via `START_HERE.md` → `STATE.md`, with `GLOSSARY.md` for terms and `MANIFEST.md` for environment/limits. The working memory lives in `STATE.md` in the repo, not in chat history. Note: `STATE.md` says it "lives in git", but this folder has **no `.git` directory** — presumably a copy/export, I cannot verify the original history.

## 2. The single most valuable next action

**Fix the emphasis-eating bug in `normalize_bullets` (`normalize.py:11`) by anchoring the bullet replacement to line starts, and add a regression test with an inline `*emphasis*` example.**

Why:

- `STATE.md` names exactly this as "The ONE next action", and the repo's own contract (`START_HERE.md` line 19) is: take the ONE action named in `STATE.md`, do it, update `STATE.md`.
- It is independently justified: the bug is **data-destroying and ships inside functionality marked "done"**. `tidymark.py` overwrites the user's file in place, so a wrong replacement silently corrupts source documents with no backup.
- I verified the bug by running the pure function (no files touched):
  - `normalize_bullets("an *important* note")` → `'an *important- note'`
  - `normalize_bullets("2 * 3 + 4 = 10")` → `'2 - 3 - 4 = 10'`

  So the blast radius is **wider than `STATE.md` describes**: not just emphasis spans, but any `* ` or `+ ` in prose or math. Also, `STATE.md`'s illustrative output (`"an -important- note"`) is slightly off — only the *closing* `*` is followed by a space, so the actual output is `an *important- note`. The bug is real; the example in the docs is imprecise and would be worth correcting while fixing.

A line-anchored fix (e.g. a `re.sub` with `^(\s*)[*+](\s)` under `re.MULTILINE`, or a per-line loop) stays inside the stdlib constraint. Preserving leading indentation matters for nested lists.

**What NOT to do** (the obvious-looking trap): `fix_headings` has a visible `# TODO: skip fenced code blocks` at `normalize.py:36` and looks nearly finished. `STATE.md` parks it explicitly and gives the reason: the hard-coded `base_level=1` would silently rewrite headings for teams whose documents start at `##`, so `base_level` must first become a documented config option (file-level or CLI). Closing that TODO or adding tests for headings is expressly forbidden as "next".

Secondary observation, not the next action: `normalize_whitespace` uses `splitlines()` + `"\n".join(...)`, which **drops the file's trailing newline** on every run. Worth logging in `STATE.md` as a follow-up rather than fixing opportunistically.

# Protocol — proj-m3 (`tidymark`)

## 1. What is this project?

A very small, single-purpose Python CLI called **tidymark** that tidies Markdown files in place.

- `tidymark.py` — argparse CLI: reads a file, applies `tidy()`, writes the result **back over the same file** (destructive, no backup, no dry-run).
- `normalize.py` — three pure functions: `normalize_whitespace`, `normalize_bullets`, `fix_headings`.
- `tests/test_normalize.py` — 2 pytest tests (whitespace, bullets). Confirmed by `.pytest_cache/v/cache/nodeids`: those two are the only tests ever collected. `fix_headings` and the CLI have **zero** test coverage.
- `NOTES.md` — an append-only running log by the owner (2026-05-02 … 2026-06-03), which is the real source of intent here.

Status by feature, per NOTES plus the code:

No git repo, no packaging (`setup.py`/`pyproject`), no CI config, no dependency manifest.

Why this and nothing else:

This is not my inference; it is a written instruction.

The bug and the prescribed fix are unaffected; I mention it because the regression test's expected value must be derived from the real behaviour, not copied from the note. Also note `+ ` has the same class of defect (e.g. `1 + 2`), and an anchored fix should cover both markers.

## 3. Constraints a contributor must respect

1. **Standard library only. No third-party dependencies at all** (NOTES 2026-06-01). Reason given: tidymark must run on bare Python in CI images and on air-gapped machines. Explicitly: **no Markdown parsing library**, "however tempting". This directly constrains the fix above — use `re` or manual line iteration, do not reach for `mistune`/`markdown-it`/`commonmark`.
2. **Do not touch `fix_headings` / heading normalization.** (NOTES 2026-05-20) It *looks* nearly finished — it runs, and there is only a small `TODO` about fenced code blocks at `normalize.py:36` — and the owner warns precisely against that reading: *"It is not."* `base_level` is hard-coded to `1`, which silently rewrites documents for teams whose headings start at `##`. He wants it unexposed **until `base_level` becomes a documented config option** (file-level or CLI flag), and states the stop was deliberate, not a matter of running out of time: *"Please leave it alone until the config question is decided."* That decision is a product question for the owner, not something a contributor should settle unilaterally. This also means the tempting fenced-code TODO is off-limits for now.
3. **Config file support and a `--check` mode are "someday" ideas**, not current scope (NOTES 2026-06-03). Note the tension: `--check` would be the natural mitigation for the destructive in-place write, but it is still parked — raise it, don't build it.
4. **Testing convention:** pytest, run as `python -m pytest` from the repo root. Tests import `normalize` as a top-level module (no package, no `conftest.py`, no `src/` layout), which is why root-dir invocation matters.

Points 1, 2 and 3 all come from `NOTES.md` only — the `README.md` mentions none of them. A contributor who reads only the README will violate all three.

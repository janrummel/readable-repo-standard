# Protocol — proj-k7 (`tidymark`)

## 1. What is this project?

`tidymark` is a small, single-folder Python CLI that tidies Markdown files **in place**. Three normalizers, all pure functions in `normalize.py`:

| Function | Does | Status per `STATE.md` |
|---|---|---|
| `normalize_whitespace` | strips trailing whitespace per line | done, tested |
| `normalize_bullets` | `* ` / `+ ` bullets → `- ` | done, tested (happy path only), **carries a known bug** |
| `fix_headings` | shifts all ATX headings so the first sits at `base_level` | **parked** (see constraints) |

`tidymark.py` is the argparse wrapper (`python tidymark.py FILE [--headings]`); it reads the file, applies `tidy()`, and writes back over the original.

The repo is deliberately built to the "Readable Repo Standard": a fresh model docks via `AGENTS.md` → `START_HERE.md` → `STATE.md`, with `GLOSSARY.md` (terms) and `MANIFEST.md` (environment/limits). Project memory lives in `STATE.md`, not in chat history. Working tree is 8 files plus `tests/`; ~40 lines of actual logic. Note: `STATE.md` says the state "lives in git", but there is **no `.git` directory** in this folder — either the snapshot was copied without it, or the repo was never initialized. I cannot tell which from here, so I am not guessing.

## 2. The single most valuable next action

**Fix the emphasis-eating bug in `normalize_bullets` — anchor the bullet replacement to line starts, and add a regression test containing inline `*emphasis*`.**

This is not my invention; it is the ONE next action `STATE.md` names, and I verified it is real by running the code:

```
normalize_bullets('an *important* note')  ->  'an *important- note'
normalize_bullets('math: 1 + 2 = 3')      ->  'math: 1 - 2 = 3'
```

Why it outranks everything else:

- **It corrupts user data silently.** `tidymark.py` overwrites the input file in place. A tool whose job is "tidy" currently mangles prose in any document containing emphasis or a `+` in arithmetic. The damage is unrecoverable without version control on the user's side.
- **It hides inside "done".** `normalize_bullets` is listed as done and has a passing test, so nothing in the test suite or the docs will surface it. `python -m pytest` is green (2 passed) with the bug live.
- **Everything else is explicitly blocked or lower-value.** Heading support is parked behind an unmade decision (below), so it is not available as an alternative. Adding features on top of a corrupting normalizer just widens the blast radius.

Concrete scope: replace the two global `str.replace` calls with a per-line transformation that only rewrites a marker in the leading-marker position (allowing leading indentation for nested lists), and add a test asserting `an *important* note` and `1 + 2 = 3` survive untouched while `* one` / `+ two` still convert.

Two secondary observations found while verifying — worth folding in only if cheap, **not** substitutes for the action above:

- `STATE.md`'s example output is slightly off: it claims `an *important* note` becomes `an -important- note`; the actual result is `an *important- note` (only the closing `*` is followed by a space). Fixing the state note while fixing the bug keeps the memory file honest.
- `normalize_whitespace` uses `splitlines()` + `"\n".join()`, so it **drops the file's trailing newline** (`'a\nb\n'` → `'a\nb'`). Since `tidymark.py` writes in place, every tidied file loses its final newline. This is undocumented and unmentioned anywhere in the repo — I am flagging it, not assuming it is intended.

## 3. Constraints a contributor must respect

1. **Standard library only, Python 3.8+.** Recorded as a decision in `STATE.md` and repeated in `MANIFEST.md`: tidymark must run on bare Python (CI images, air-gapped machines). Explicitly: **do not add a Markdown parsing library**, no `pip install` dependencies. This directly shapes the fix — the bullet anchoring must be done with `re`/string slicing, not a parser.
2. **`fix_headings` and `--headings` are PARKED. Do not touch them.** `STATE.md` warns that this area *looks* finished (the code runs; there is only a small `# TODO: skip fenced code blocks` at `normalize.py:36`) and pre-emptively forbids the obvious moves: do not add tests for it, do not close that code-block TODO, do not expose it further. The reason is recorded: the hard-coded `base_level=1` would silently rewrite headings for teams whose documents start at `##`. `base_level` must first become a documented config option (file-level or CLI) — that is a design decision the owner has reserved, not a coding task.
3. **Work only through `STATE.md`.** The stated workflow (`START_HERE.md`): take the ONE next action named in `STATE.md`, do it, then update `STATE.md`. State is the deliverable alongside the code — a fix that leaves `STATE.md` stale is half done.
4. **The tool rewrites files in place** with no backup and no `--dry-run`. Any change to normalizer behavior is directly a change to user data; test on copies.

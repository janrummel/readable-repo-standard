# Protocol — proj-k7 (`tidymark`)

## 1. What is this project?

`tidymark` is a tiny standard-library-only Python CLI that tidies Markdown files in place:

- `normalize_whitespace` — trims trailing whitespace per line
- `normalize_bullets` — converts `* ` / `+ ` list bullets to `- `
- `fix_headings` — shifts all ATX heading levels so the first heading sits at `base_level` (behind the `--headings` flag)

Nine files, ~36 KB. `normalize.py` holds the pure functions, `tidymark.py` is the argparse wrapper that reads a file, tidies it, and overwrites it. `tests/test_normalize.py` has 2 tests; both pass (`python3 -m pytest` → `2 passed`).

The repo explicitly follows the "Readable Repo Standard": memory lives in git, not in chat. `START_HERE.md` → `STATE.md` is the documented entry path, with `GLOSSARY.md` (terms), `MANIFEST.md` (environment/limits) and `AGENTS.md` (agent docking bridge) as supporting docs.

## 2. Single most valuable next action

**Fix the emphasis-eating bug in `normalize_bullets` (`normalize.py:11`) by anchoring the replacement to line starts, and add a regression test with an inline `*emphasis*` example.**

Why this and nothing else:

- **The repo names it explicitly.** `STATE.md` has a section literally titled "The ONE next action" and this is it. There is no ambiguity to resolve.
- **I verified the bug is real**, not stale documentation. Running the current code in memory:
  - `normalize_bullets("an *important* note")` → `'an *important- note'`
  - `normalize_bullets("a + b = c")` → `'a - b = c'`
- **It is corruption inside shipped, "done" functionality.** `tidymark.py` writes results back over the input file, so this silently damages users' prose on every run. That outranks any new feature.

Two implementation details worth carrying into the fix, which I found by reading the code rather than the docs:

- A naive `^\* ` anchor would regress indented bullets. `normalize_bullets("  * indented bullet")` currently works correctly (`'  - indented bullet'`); the anchor must allow leading whitespace, e.g. `re.sub(r"(?m)^(\s*)[*+] ", r"\1- ", text)`. The existing happy-path test does not cover indentation, so this regression would pass CI unnoticed.
- **Small correction to `STATE.md`:** it predicts `"an *important* note"` becomes `"an -important- note"`. The actual output is `"an *important- note"` — the opening `*` is followed by a letter, not a space, so only the closing `*` matches. The bug is real and the priority is right; only the illustrative example in the state file is slightly off. Worth correcting when updating `STATE.md`.

**Explicitly not the next action** — the `fix_headings` / `--headings` work. It looks nearly finished (the code runs, one small `# TODO: skip fenced code blocks` at `normalize.py:36`, and I confirmed the TODO describes a real defect: a `#` inside a fenced block does get shifted). But `STATE.md` parks it with a stated reason: the hard-coded `base_level=1` is wrong for teams whose documents start at `##`, and shipping it would silently rewrite their headings. `base_level` must first become a documented config surface. This is a deliberate trap for a fresh reader, and I am flagging it rather than taking the bait.

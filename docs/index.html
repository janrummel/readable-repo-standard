Readable Repo Standard · cold-start test

Put the agent harness
in the repo.

Every agent harness is built differently. So instead of wrapping the model in a harness, the harness minimum, the brain, the memory, the vocabulary, lives in the repo itself. Then any blank model opens the folder and just knows what to do next. And a pre-registered test measured whether it holds up.

the repo is the book · the model is only the pen · a pre-registered A/B experiment
The problem

A five-minute pause and a three-month pause look identical to a model

It starts from zero every time. And after a long gap, you have forgotten the state too. If the "what's done, what's next, and why" lived only in the last conversation, it is gone. The next agent has to guess, and it fills every gap with a confident guess.

The test

One project, two repos, identical code

A tiny synthetic tool. The only difference between the two copies: one has four small files that record the state, plus the bridge that points an agent at them (START_HERE · STATE · MANIFEST · GLOSSARY · AGENTS). Blank agents were dropped into each, cold, and asked for the single correct next action. The project had a trap: a feature that looked finishable but was deliberately parked, while the real next task was a subtle bug whose priority lived only in STATE.md.

Every bare agent confidently did the wrong thing: it worked on the feature that looked finishable instead of the real priority. The readable repo let every agent take the correct action, refuse the parked feature, and name the hidden rule.
Why it happened

The bare agent is not less capable. It is uninformed, and confidently so.

One bare agent even spotted the real bug in the code, and dismissed it anyway, because nothing in the repo told it that was the priority.

Seeing the bug was not enough.
Knowing it was next required the recorded state.
How it works

Four files at the root, plus a short bridge

No framework, no runtime, no tool lock-in. One external rule: open START_HERE.md first. From there the repo carries itself.

START_HERE.mdThe entry point. What this is, a map, where to go next.
STATE.mdThe memory. The one next action, what's done, what's parked, decisions with why.
MANIFEST.mdHow to run it and what it needs.
GLOSSARY.mdEvery project-specific term, defined once.
AGENTS.mdA short bridge that points any agent to START_HERE.
Honest boundaries

What this does not claim

The result is real, and it is bounded. The point of a pre-registered test is to report the limits as loudly as the wins.

a plain notes file scored the same: 3/3 n = 5 per arm, one synthetic project earlier runs graded by the author, unblinded confidence is self-reported, not calibrated good code comments shrink the gap RRS complements code hygiene, not replaces it

The first chip is the important one. A third variant put the four files against the control nobody had run yet: the same code, the same facts, written as one unstructured NOTES.md. It scored the same. So what carries the effect is recording the state where the next reader will find it, not this particular set of files. Where structure is meant to earn its keep is what the test cannot reach: a bigger project, a note grown over a year, several authors, and the moment two notes have to be merged. That is the bet, and it is a bet, not a result. Predictions and grading rules were committed before the runs, grading was blind to the arm, and the raw runs are in the experiment folder.
